## THE ONE PAYMENT RAIL (the single standard)

Every product — Wallet, Deploy, Crash, and every external merchant — is a **consumer of this one rail**. The Wallet is just the premium consumer app of it. **The rail is x402** — vanilla **x402 V2 'exact'** over Sui's protocol-level **GASLESS Address-Balance transfers**. There are **no custom Move verbs** in the payment path: the payer signs a `send_funds` PTB with its **own key** (`gasPayment=[]`, `gasPrice=0` — no gas token, ever), and the Suize facilitator settles it KEYLESS over gRPC. *"Your address is your account"* is literal on-chain — no Account object, no fee object. (The only Suize Move code in the payment path is the standalone **subscription** module, below; `deploy_sui` + `crash` are merchant-side, not the rail.)

### How a payment works (no verbs, just x402 + a fee output)

A merchant declares a 402 challenge (the x402 `PaymentRequirements`); the payer builds the gasless transfer and signs the bytes; the facilitator verifies + settles. The **2% (with a $0.01 minimum) fee is MERCHANT-ABSORBED** — carved as a **second declared output** (`extra.outputs`) inside the SAME atomic gasless transaction, enforced by the facilitator at verify time. So:

- **The on-chain balance-change set IS the receipt** — fee visible, no separate receipt event, no on-chain fee object. The fee output lands at the Suize treasury (resolved from the `treasury@suize` SuiNS name).
- **No free tier — the fee is NEVER waived** (owner law 2026-06-14): EVERY payment carries the 2% (with a $0.01 minimum). An unregistered merchant pays the default rate; the `SUIZE_MERCHANTS` registry only CUSTOMIZES it. The only single-output result is structural (a first-party merchant == treasury, or a sub-unit amount), never a tier. The payer **always pays exactly the listed price**.
- Idempotency rides the x402 `payment-identifier` extension — **never** an ad-hoc field; a retry is never a double charge.

### The facilitator surface (`api.suize.io`; `facilitator.suize.io` alias) — STATELESS

> **Status (calibrated honesty):** the x402 facilitator is **BUILT + E2E-PROVEN on real testnet** (21/21 E2E green) — endpoints + wire (the merged x402 V2 Sui exact scheme): `services/backend/SPEC.md` §7 (reference, never redeclare). **The chain is the database:** Suize stores nothing per payment; `/verify` and `/settle` are one tx read/write each. Webhooks, checkout-session stores, server-minted ids are **GONE**.

- **The endpoints:** `/verify`, `/settle` (keyless, idempotent), `/supported`, `/build`, `/terms`, `/tx`. The facilitator runs the live x402 service for Sui.
- **TWO PAYER DOORS, ONE WIRE (owner-locked 2026-06-15 — humans out of scope):** the agent signs the gasless payment **itself** and presents `X-PAYMENT`. (1) **Sui-aware** — its own Sui key + USDC; *"your address is the account."* (2) **Suize** — its Suize zkLogin session via **`@suize/mcp`** (`claude mcp add suize -- npx -y @suize/mcp`, Google sign-in once; `suize_pay` reads any merchant's 402 challenge, enforces the confirm dials, signs with the **LOCAL** session, settles via the facilitator — the MCP is the GENERIC payment wallet, NOT a per-product toolkit: Deploy is reached through this same x402 wire, never a bespoke tool). There is **NO human / pay-link / relay path** — the pay-link/pickup/hosted-pay-page system is **DELETED**.
- **Merchant integration = the `@suize/pay` middleware (~60 lines)** — zero wallet/chain code merchant-side (this is what makes "one line, live in minutes" literally true): answer the **402** x402 challenge; on the retry, ONE call against the merchant's OWN terms → serve. The merchant is the verifier; Suize is plumbing. (Platform gateway plugins are ROADMAP — no platform names in public copy until one ships.) **There is no hosted pay page** (`apps/pay`/`pay.suize.io` and the SSO bridge are gone): an agent pays the merchant's own x402 endpoint directly.

### Subscriptions — a standalone on-chain Party-object module (PUSH, not pull)

The recurring half of the rail is its OWN Move module (`subs::subscription`, published testnet) — separate from the x402 one-off path (which has no Move code at all). The user creates a **`Subscription` object** ONCE (fixed merchant + amount + period), paying the first period inline. **Every renewal is a USER-SIGNED, gas-sponsored transaction that pushes exactly one period** — nothing ever reaches INTO the user's funds (the object holds no balance; the caller pushes one period's coin in, the module asserts `value == amount`, carves the 2%/$0.01 to the treasury, sends the rest to the merchant). The object is **soulbound** (`key`, no `store`); **cancel = deleting the object on-chain.** Merchants **self-index subscription state from chain events** (Suize stores nothing). The user's wallet renews silently when open; otherwise a reminder. The off-chain relayer **never holds a key** — it only SPONSORS gas; the chain itself is the double-charge guard (a second in-window renewal aborts `ETooEarly`).

### Two pots, two control layers

- **Two pots** *(internal concept term — consumer copy says **"sub-account"**, see the vocabulary laws below)*: the human's own USDC (Suize never touches it) vs the **detached agent's address** (the agent's allowance; **its balance = the hard cap**). Funding = the human sends USDC to that address; a detached agent is a SEPARATE Google sign-in's address.
- **Two control layers:** (1) **on-chain physics** — the funded balance is the hard cap, the human can stop funding + sweep instantly, subscription terms (fixed payee + per-period amount) are the recurring leash; (2) **client-side policy dials** in the PAY app (or the dev-side MCP) — confirm-each / auto-under-$X / full-auto / confirm-new-subscription. **Subscriptions, once approved, renew silently (push, user-signed).**

### Custody

**Fully non-custodial by construction** — the PAY app (or, on the dev CHARGE-side, the local MCP) runs Google/Enoki zkLogin and signs locally; keys never leave the user's machine; Suize never signs owner txs and never holds the user's key. **No on-chain agent identity, no delegated agent key** — a detached agent is just a second sign-in's address. Honest caveat: that funded balance is **delegated-spend** (v1 has no payee allow-list) — bounded by what you fund + a verifiable log + stop-funding-and-sweep. **Delegated-spend risk, NOT custody risk.**
